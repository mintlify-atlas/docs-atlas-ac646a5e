## Why Trix?

Trix takes a fundamentally different approach to rich text editing. Rather than wrapping HTML's `contenteditable` and `execCommand` APIs — which are inconsistently implemented across browsers — Trix treats `contenteditable` as an I/O device.

When input reaches the editor, Trix converts it into an operation on its internal document model, then re-renders that document back into the editor. This gives Trix complete control over every keystroke and eliminates browser inconsistencies entirely.

## Quick example

Add Trix to any web page with just a script tag and two elements:

```html
<head>
  <link rel="stylesheet" href="https://unpkg.com/trix@2.1.17/dist/trix.css">
  <script src="https://unpkg.com/trix@2.1.17/dist/trix.umd.min.js"></script>
</head>
<body>
  <input id="content" type="hidden" name="content">
  <trix-editor input="content"></trix-editor>
</body>
```

That's it. Trix automatically creates a toolbar, wires up the hidden input, and the editor is ready to use.

## Built on web standards

Trix is built with established web standards:

- [Custom Elements](https://developer.mozilla.org/en-US/docs/Web/Web_Components/Using_custom_elements) — `<trix-editor>` and `<trix-toolbar>` are native web components
- [Element Internals](https://developer.mozilla.org/en-US/docs/Web/API/ElementInternals) — for native form association
- [Mutation Observer](https://dom.spec.whatwg.org/#mutation-observers) — for efficient DOM change detection
- [DOMPurify](https://github.com/cure53/DOMPurify/) — for HTML sanitization

Trix supports all evergreen desktop and mobile browsers.
